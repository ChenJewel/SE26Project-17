# 01 产品范围

## 产品定位

ueat 是校园约饭与饭后社区原型。当前目标不是完整商业化产品，而是先验证核心流程：

- 找到合适的饭搭子。
- 发布自己的约饭卡。
- 通过社区内容了解用户和校园生活。
- 通过聊天交换约饭意向。
- 在“我的”页管理偏好、头像、内容和设置。

## 核心角色

- 当前用户：浏览首页卡片、发布约饭卡、发帖子、聊天、管理个人信息。
- 其他用户：出现在约饭卡、社区帖子、搜索结果、聊天会话中。
- 系统：负责推荐卡片、生成交换卡片消息、展示匹配与状态。

## 当前核心流程

1. 首页浏览约饭卡。
2. 多选标签筛选卡片。
3. 左右滑卡切换候选卡片。
4. 点击“想一起吃”进入聊天详情，系统发送交换约饭卡。
5. 双方选择“拒绝”或“聊聊看”。
6. 在社区浏览、发布、查看图文/视频帖子。
7. 搜索用户、约饭卡片和帖子，并打开详情。
8. 在“我的”页编辑偏好标签、头像，查看自己的内容与互动记录。

## 当前原型边界

- 没有真实后端，所有数据存在 React state 或 mock 文件中。
- 没有真实用户系统，用户目前以昵称和头像字符标识。
- 没有真实图片/视频上传，媒体展示多为 CSS 视觉占位。
- 没有真实消息推送，交换卡片请求保存在本地 state。
- 没有真实路由系统，页面由 `App.tsx` 的 `currentPage` 控制。

## 后续需要产品确认的问题

- 匹配算法是否以标签为主，还是还要考虑时间、地点、聊天偏好、共同好友等。
- “想一起吃”是单向邀请、双向确认，还是先交换卡片再决定。
- 用户主页对陌生人开放哪些内容。
- 社区内容和约饭卡是否需要审核、举报、黑名单联动。
- Android App 展示版是否需要补充原生能力，例如状态栏、返回键、图片上传、消息推送。
