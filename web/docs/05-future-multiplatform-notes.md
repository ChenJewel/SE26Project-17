# 05 多端迁移注意事项

当前项目是 React + Vite Web 原型。它适合快速验证交互，但不能直接发布为小程序、App 或鸿蒙应用。后续迁移时建议保留原型作为参考，再新建多端工程或逐步抽共享层。

## 推荐迁移节奏

1. 继续在 Web 原型中完成主要页面和交互。
2. 稳定后抽出共享类型、mock 数据、业务规则。
3. 新建 Taro/uni-app 或其他多端工程。
4. 先迁移核心页面：首页、发卡片、消息、我的、设置。
5. 再迁移社区和复杂媒体交互。
6. 最后适配平台能力：图片上传、视频播放、消息推送、定位、登录。

## 可复用部分

- 产品流程。
- 页面结构。
- 数据类型。
- 文案。
- 标签、约饭卡、帖子、交换请求等业务概念。
- 部分 React 组件拆分思路。

## 需要重写或适配的部分

- DOM 标签和浏览器事件。
- Tailwind/Web CSS 细节。
- Pointer event 滑卡手势。
- `window.scrollTo` 等浏览器 API。
- Web 浮层层级和 fixed 布局。
- lucide 图标在小程序/鸿蒙中的资源形式。
- 图片/视频真实媒体能力。

## Taro 方向建议

如果选择 Taro：

- `div/button/input` 等需要改成 Taro 组件或兼容写法。
- 路由应使用 Taro 页面配置。
- 首页滑卡手势要改成 Taro 触摸事件。
- CSS 需要按小程序限制整理，避免复杂动态 class。
- App 端如果走 React Native，需要额外确认组件和样式支持度。

## 鸿蒙方向建议

鸿蒙端通常需要独立 ArkTS/ArkUI 工程。可复用：

- 数据模型。
- 接口协议。
- 页面流程。
- 交互文档。
- 设计规范。

不建议直接复用 Web UI 代码。

## 后端/接口建议

后续至少需要这些接口或服务：

- 用户与认证。
- 用户主页与偏好标签。
- 约饭卡发布、列表、详情、筛选。
- 社区帖子发布、列表、详情。
- 评论、点赞、收藏、举报。
- 搜索。
- 聊天会话。
- 交换约饭卡请求。
- 媒体上传与资源访问。

## 当前原型中特别要动态化的地方

- 用昵称匹配用户改为 `userId`。
- 搜索详情和社区详情合并。
- 交换卡片请求从本地 state 改为后端消息。
- 图片/视频从 CSS 占位改为真实媒体资源。
- 设置页可根据后端用户配置和系统能力动态展示。
