# 14 首页饭卡匹配算法现状与评估

本文记录当前首页饭卡匹配算法的线上代码现状、评分因子、数据来源、前端筛选关系，以及对算法合理性、隐患和后续改进方向的评估。

结论先行：当前算法是一个可解释的启发式多因子排序算法，不是机器学习模型，也不是 embedding/向量召回。它适合当前数据量小、需要快速稳定上线的阶段，但还没有解决语义归一、真实用户反馈学习、冷启动质量、时间地点结构化和效果评估问题。

## 当前入口

### 后端排序入口

首页饭卡列表来自：

- 前端：`web/src/hooks/useMealCards.ts`
- API：`GET /meal-cards`
- 后端路由：`server/src/modules/mealCards.ts`
- 排序核心：`server/src/modules/recommendation.ts`

`GET /meal-cards` 的当前流程：

1. 读取当前用户 ID：`getCurrentUserId(req)`。
2. 查询所有 active 饭卡：`postgresStore.listActiveMealCards()`。
3. 查询当前用户资料：`postgresStore.findUserById(currentUserId)`。
4. 如果当前用户不存在，直接返回原始饭卡列表，不做个性化排序。
5. 如果当前用户存在，补充作者资料、拉黑关系、举报计数和约饭请求历史。
6. 调用 `rankMealCardsForUser(cards, context)`。
7. 返回已经过滤、重排，并写入动态 `matchScore` 和 `reason` 的饭卡列表。

### 前端展示入口

前端 `useMealCards` 只负责拉取和维护饭卡列表。首页 `Home.tsx` 直接展示后端返回的：

- `card.matchScore`
- `card.reason`

也就是说，首页显示的匹配百分比和匹配理由主要由后端 `recommendation.ts` 决定。

## 后端硬过滤

排序前先排除不应该推荐的饭卡。

当前过滤规则在 `isRecommendableCard`：

| 条件 | 结果 |
| --- | --- |
| `card.status !== "active"` | 不推荐 |
| 饭卡作者是当前用户自己 | 不推荐 |
| 当前用户拉黑了作者 | 不推荐 |
| 作者举报数达到阈值 | 不推荐 |
| 饭卡举报数达到阈值 | 不推荐 |

当前举报硬过滤阈值：

```text
highRiskReportThreshold = 5
```

注意：当前 `listBlockedUserIdsFor` 只查“当前用户拉黑的人”，不包含“对方拉黑当前用户”的反向关系。如果产品希望双向不可见，需要扩展这里。

## 总分公式

当前总分会被限制在 `0-1`，最后换算为 `1-99` 的整数百分比：

```text
score =
  reciprocalScore * 0.25 +
  sceneScore      * 0.22 +
  interestScore   * 0.20 +
  behaviorScore   * 0.13 +
  trustScore      * 0.08 +
  qualityScore    * 0.07 +
  freshnessScore  * 0.03 +
  explorationScore* 0.02 -
  penalty

matchScore = clamp(round(score * 100), 1, 99)
```

当前权重：

| 因子 | 权重 | 当前含义 |
| --- | ---: | --- |
| `reciprocalScore` | 25% | 双向偏好可能性。既看我是否喜欢这张卡，也粗略估计对方是否可能接受我。 |
| `sceneScore` | 22% | 吃饭场景契合度，主要看饭点、地点、人数/氛围和新鲜度。 |
| `interestScore` | 20% | 兴趣/偏好相似度，主要看用户偏好标签、饭卡标签、地点、人数说明、作者资料。 |
| `behaviorScore` | 13% | 历史约饭请求行为，接受加分，待处理小幅加分，被当前用户拒绝扣分。 |
| `trustScore` | 8% | 作者资料可信度和举报风险。 |
| `qualityScore` | 7% | 饭卡内容完整度。 |
| `freshnessScore` | 3% | 发布新鲜度。 |
| `explorationScore` | 2% | 稳定哈希探索分，让低历史分卡片有一点曝光机会。 |
| `penalty` | 扣分 | 举报、当前用户已拒绝等负反馈。 |

这个公式说明当前算法更偏“约饭意图匹配”，不是单纯“兴趣社交匹配”。前三项合计 67%，所以核心排序由双向偏好、场景和兴趣决定。

## 各因子细节

### 1. reciprocalScore：双向偏好

数据来源：

- 当前用户 `preferenceTags`
- 作者 `preferenceTags`
- 饭卡 `tags`
- 相关 exchange requests

计算方式：

1. `myInterestToCard` 直接复用 `interestScore`。
2. `targetLikelyAcceptMe` 使用作者偏好标签 + 饭卡标签，与当前用户偏好标签做 Jaccard 相似度。
3. `pairCore = min(myInterestToCard, targetLikelyAcceptMe)`。
4. 如果双方已有 accepted/pending 约饭请求，额外加分。

当前问题：

- 这不是严格双向匹配，因为没有当前用户自己的饭卡、可约时间、位置距离等结构化信号。
- 作者偏好和当前饭卡标签混在一起，会把“这顿想吃什么”和“这个人长期喜欢什么”混为一谈。

### 2. sceneScore：场景匹配

由四部分组成：

```text
sceneScore =
  timeScore         * 0.35 +
  placeScore        * 0.35 +
  peopleScore       * 0.20 +
  availabilityScore * 0.10
```

当前实现：

- `timeScore`：识别“今天、今晚、中午、明天”和简单小时数字。
- `placeScore`：用地点文本和用户偏好标签做包含关系，并对“附近、同校、食堂、随便、都可以”给基础分。
- `peopleScore`：识别“都可以、随便、1、一、2、3”等文本。
- `availabilityScore`：复用新鲜度。

当前问题：

- 时间是自由文本，不知道真实日期、开始时间、过期状态。
- 地点是自由文本，不知道校区、距离、食堂 ID、经纬度。
- 人数/氛围也是自由文本，不知道“1 人”“2-3 人”“社恐友好”等稳定结构。

### 3. interestScore：兴趣和偏好相似

由四部分组成：

```text
interestScore =
  tagOverlapScore     * 0.50 +
  textOverlapScore    * 0.20 +
  profileOverlapScore * 0.20 +
  communityTopicScore * 0.10
```

当前实现：

- `tagOverlapScore`：当前用户偏好标签 vs 饭卡标签 + 地点 + 人数说明，Jaccard。
- `textOverlapScore`：用户偏好标签是否被饭卡文案、地点、人数说明、标签包含。
- `profileOverlapScore`：当前用户偏好标签 vs 作者偏好标签 + 学校 + bio。
- `communityTopicScore`：取标签重合和文本重合的较高值再乘 0.7。

当前问题：

- 只做字符串归一和包含，没有真正的中文分词、同义词、embedding 或分类器。
- “不吃辣”和“清淡”只有完全写到标签/文案里才容易命中。
- Jaccard 对短标签非常敏感，用户只填 1-2 个标签时分数容易跳动。

### 4. behaviorScore：历史互动

当前只使用 `exchange_requests`。

规则：

- 没有历史请求：默认 0.45。
- accepted：加 0.30。
- pending：加 0.08。
- rejected：扣分，当前用户发起后被拒绝扣得更多。

当前问题：

- 没有使用点击、滑过、停留、私聊、已读、关注、收藏等行为。
- 没有区分最近一次行为和很久以前的行为。
- repeated exposure 没有处理，同一张卡可能被用户看过多次还继续出现。

### 5. trustScore：信任与安全

加分项：

- 饭卡或作者 verified。
- 作者 profileCompleted。
- 作者有头像。
- 作者 bio 足够长。

扣分项：

- 用户举报数。

当前问题：

- 当前只把举报数作为粗粒度风险信号。
- 没有机器审核、内容安全分类、封禁状态、学校认证强弱、手机号/邮箱可信级别。
- 资料完整度不一定等于可信度，容易被刷。

### 6. qualityScore：内容质量

加分项：

- 文案长度达到 12/28 字。
- 标签数至少 2 个。
- 时间、地点、人数都填写。
- 有媒体。

扣分项：

- 标签超过 6 个会小幅扣分。

当前问题：

- 内容质量仍是表面特征。
- 没有识别低质量文案、广告、重复内容、复制粘贴、敏感内容。

### 7. freshnessScore：新鲜度

当前按 `createdAt` 距离当前时间计算：

| 年龄 | 分数 |
| --- | ---: |
| 6 小时内 | 1.00 |
| 24 小时内 | 0.86 |
| 72 小时内 | 0.64 |
| 168 小时内 | 0.42 |
| 更久 | 0.25 |
| 无创建时间 | 0.45 |

当前问题：

- 它看的是发布时间，不是饭局时间。
- 一张“明天中午”的卡如果发布时间较早，可能被低估。
- 一张刚刚发布但约饭时间已经过了的卡，可能仍被新鲜度加分。

### 8. explorationScore：探索

当前根据饭卡 ID 做稳定 hash：

```text
0.35 + (hash % 40) / 100 + lowStoredScoreBonus
```

如果饭卡原始存储 `matchScore < 75`，会加一点探索补偿。

当前问题：

- 这不是真正的多臂老虎机或流量探索。
- 没有曝光计数，也没有“探索后用户是否喜欢”的反馈闭环。
- 因为 hash 稳定，同一用户看到的探索扰动长期固定。

## 推荐理由生成

`buildReason` 会基于分项分数生成最多 2 条中文理由。

优先级大致是：

1. 场景强匹配：饭点、地点、人数接近。
2. 饭点适合今天直接约。
3. 共享标签。
4. 双方偏好有交集。
5. 已有积极互动。
6. 对方资料完整可信。
7. 卡片内容完整且有媒体。
8. 新卡。
9. fallback：标签和文案与你的偏好有重合，或原始 `card.reason`。

当前问题：

- 理由是模板生成，不一定完整解释最终排序。
- 只展示 2 条，可能掩盖真正拉开分数的扣分项。
- 没有返回 debug 分项，线上排查为什么某张卡靠前/靠后不方便。

## 首页顶部标签筛选

首页顶部筛选在前端 `Home.tsx`，不属于后端匹配排序。它是在后端排序结果上再做本地过滤。

当前实现：

1. 用户选择一个或多个筛选标签。
2. 对每张饭卡调用 `cardMatchesFilter(card, tag)`。
3. 饭卡需要满足所有已选标签的筛选条件。
4. 筛选条件是：
   - 饭卡标签、文案、地点、人数、推荐理由、昵称、时间拼成文本。
   - 归一化后检查是否包含筛选词。
   - 如果没有直接包含，再检查 `relatedTerms` 里配置的少量别名。

当前 `relatedTerms` 例子：

| 筛选词 | 别名 |
| --- | --- |
| 晚饭 | 晚餐、晚上、今晚、18:、19: |
| 午饭 | 午餐、中午、12: |
| 安静 | 自习、图书馆、少说话、不吵 |
| 聊天 | 聊、话题、交流 |
| 健身 | 运动、跑步、训练 |
| 清淡 | 不辣、少油、轻食 |

重要边界：

- 筛选不会重新计算 `matchScore`。
- 筛选不会调用后端。
- 筛选是硬过滤，不是按相关性重排。
- 多个筛选标签之间是 AND 关系。

这和“筛选标签不是必须等于饭卡 tag”这个产品要求是一致的，但当前只做到“包含 + 少量手写别名”，还没有做到真正语义模糊搜索。

## 合理性评估

### 合理的地方

1. 适合当前阶段：数据少时，启发式规则比训练模型更可控。
2. 可解释：每个因子能对应到用户可理解的理由。
3. 有安全底线：自己、拉黑、举报高风险内容会被过滤。
4. 约饭场景优先：时间、地点、人数权重合计足够高，不会只按兴趣推荐。
5. 有冷启动兜底：用户标签少、历史行为少时仍有默认分，不会全是 0。
6. 不依赖 AI 服务：核心 `/meal-cards` 不会被模型生成、embedding 服务或异步任务拖慢。

### 不足和隐患

1. 权重没有数据验证。
   目前 25/22/20/13 等权重是工程经验值，没有 A/B test、点击率、邀请率、接受率或用户满意度验证。

2. 文本语义能力弱。
   当前只做 trim/lowercase/去空格、包含和 Jaccard。中文同义词、近义表达、错别字、口语表达不能稳定匹配。

3. 时间地点没有结构化。
   饭局时间、发布日期、地点文本混在一起，无法准确判断“是否快过期”“是否离我近”“是否同校区”。

4. 双向匹配只是近似。
   `reciprocalScore` 没有使用当前用户自己的饭卡、可约时间、距离、性别/安全偏好、主动屏蔽偏好等更强信号。

5. 行为反馈闭环不足。
   现在只看 exchange request，没有看曝光、跳过、点击详情、停留、私信、关注、拉黑、举报、接受邀约后的真实到店/完成状态。

6. 新卡和老卡平衡还粗糙。
   新鲜度权重只有 3%，探索 2%。新用户/新饭卡可能仍然难以获得足够曝光，也没有去重曝光策略。

7. 举报和信任策略粗糙。
   举报数达到 5 才硬过滤，低于阈值只扣分。没有按举报类型、严重程度、处理状态、时间衰减做区分。

8. 前端筛选和后端排序割裂。
   后端给出个性化排序，前端又本地 AND 过滤。筛选后不会重新排序，也不会让后端知道用户当前明确意图。

9. `matchScore` 易被误解为真实概率。
   当前百分比只是启发式归一化分，不代表“成功约饭概率”或“互相喜欢概率”。

10. 缺少可观测性。
    目前没有记录每张卡的分项分、排序位置、曝光、点击、邀请、接受、跳过等日志，后续很难判断算法是否变好。

11. 缺少测试覆盖。
    推荐算法没有专门单元测试。权重或阈值被改动后，无法自动发现排序回归。

12. 性能会随饭卡数线性增长。
    当前 `GET /meal-cards` 先查所有 active 饭卡再在 Node 内存里排序。小规模可用，但 10k+ active 饭卡时需要候选召回、分页和数据库预过滤。

## 当前最应该补的能力

### P0：不改变架构的小修

1. 给 `recommendation.ts` 增加单元测试，覆盖过滤、权重、理由、举报、历史请求等场景。
2. 返回或内部记录 debug 分项，至少在开发环境能看到每张卡为什么得分。
3. 把饭局时间拆成结构化字段，例如 `mealDateTime`、`expiresAt`。
4. 前端筛选改为相关性打分和重排，而不是只做 AND 硬过滤。
5. 明确 `matchScore` 的文案，不要暗示它是成功概率。

### P1：推荐效果闭环

1. 增加曝光、滑过、点击详情、邀请、接受、拒绝、拉黑、举报等行为日志。
2. 做离线评估指标：
   - 邀请率
   - 接受率
   - 私聊开启率
   - 举报率
   - 拉黑率
   - 新用户首日成功互动率
3. 根据行为日志调权重，而不是凭感觉调。
4. 增加 repeated exposure 控制，看过多次未互动的饭卡降低曝光。

### P2：语义匹配升级

1. 建立标签归一表，例如“少辣/不辣/清淡”“二食堂/二餐/第二食堂”。
2. 对饭卡文案、用户偏好、公开主页内容做轻量 embedding。
3. 用“规则召回 + 向量召回 + 规则重排”的混合方案。
4. 大模型只用于生成自然语言理由或标签归一建议，不直接决定排序。

2026-07-20 补充决策：AI 破冰模块已经有 `ai_memory_items`、`user_ai_profiles`、canonical tags、`embedding_json` 和 pgvector 可选召回，但当前 embedding 仍是 `local-hash-embedding-v1`。首页匹配下一步不应单独另起一套语义系统，而应复用 AI 破冰升级后的共享语义层。

推荐迁移方式：

```text
AI 破冰先从 local-hash-embedding-v1 升级到专用 embedding 模型
-> 抽出 server/src/modules/semanticSignals.ts
-> recommendation.ts 读取 canonical tags / 用户画像 / 饭卡向量
-> semanticScore 并入 interestScore
-> 缺失向量时回退当前规则匹配
```

首页 `/meal-cards` 仍不能同步等待大模型。可以在后台慢慢生成饭卡和用户画像 embedding，但首页打开时必须读取已有缓存或降级到当前启发式排序。详细计划见 [15-semantic-embedding-upgrade-plan.md](./15-semantic-embedding-upgrade-plan.md)。

## 推荐路线

短期不要直接上复杂模型。更稳的路线是：

```text
M1 当前规则排序稳定化
  -> 单元测试
  -> debug 分项
  -> 时间地点结构化
  -> 前端筛选相关性排序

M2 行为日志和离线评估
  -> 记录曝光/点击/邀请/接受/拒绝/举报
  -> 建立基础指标看板
  -> 用数据调权重

M3 语义召回
  -> 标签归一
  -> 专用 embedding 模型
  -> 复用 AI 破冰的 ai_memory_items/user_ai_profiles
  -> 规则召回 + 向量召回

M4 学习排序
  -> 使用真实行为训练轻量排序模型
  -> 保留安全过滤和规则兜底
```

这样做的好处是，当前算法不会被推倒重来，后续只是在候选召回、特征质量、权重学习和理由生成上逐步增强。
