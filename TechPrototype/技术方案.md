# ueat 第二周技术方案

## 1. 技术目标

本轮技术原型从第一周界面原型继续推进，目标是验证 ueat 从“可点击前端原型”走向“可安装 Android App 技术原型”的关键技术路径。

重点验证内容：

1. 约饭卡、社区帖子、评论、关注、聊天等核心数据可以从本地 state 迁移到后端接口与数据库。
2. “想一起吃”可以生成交换约饭卡请求，并进入可持续演进的会话模型。
3. 推荐匹配逻辑有明确输入、输出和可解释理由，能够支撑首页卡片排序与筛选。
4. 系统架构能支持 Android App 展示，并为后续多端扩展保留接口边界。
5. 技术风险可识别、可测试、可缓解。

## 2. 架构风格选择

### 2.1 总体架构

采用分层架构 + 前后端分离 + 模块化服务的架构风格。

推荐分层：

| 层次 | 职责 | 原型阶段对应 |
| --- | --- | --- |
| 表现层 | 移动端 Web 页面、交互、表单、弹层、路由 | `web/src/pages`、`web/src/components` |
| App 容器层 | Android WebView、APK 打包、原生权限扩展 | Capacitor Android |
| 状态与业务协调层 | 页面状态、跨页面跳转、业务动作编排 | `web/src/hooks`、`web/src/App.tsx` |
| API 服务层 | REST API、鉴权、业务校验、聚合查询 | 第二周技术原型新增后端 |
| 实时消息层 | 聊天消息、交换卡片状态、已读事件 | WebSocket 或托管实时服务 |
| 数据持久层 | 用户、约饭卡、帖子、评论、互动、消息 | 关系型数据库 |
| 媒体资源层 | 图片、视频上传、访问、审核扩展 | 对象存储或本地 mock |

### 2.2 选择理由

- 当前前端已经形成页面、组件、hooks、types、data 的初步边界，适合平滑迁移到分层架构。
- 约饭、社区、聊天、通知等模块边界清晰，分层架构便于团队协作开发。
- REST API 适合约饭卡、帖子、评论、用户资料等常规业务；WebSocket 适合聊天和交换请求状态同步。
- MVP 阶段避免过早引入微服务，降低部署和协作复杂度。

## 4. 架构视图设计

### 4.1 逻辑视图

系统按领域模块拆分：

- 用户与认证模块：注册、登录、校园邮箱认证、用户资料。
- 约饭卡模块：发布、浏览、筛选、详情、关闭/删除。
- 推荐匹配模块：标签、时间、地点、认证状态、活跃度综合评分。
- 社区模块：帖子、媒体、评论、点赞、收藏。
- 社交关系模块：关注、拉黑、举报。
- 聊天模块：会话、消息、约饭卡交换、状态同步。
- 通知模块：赞藏、评论、关注、系统消息。

### 4.3 进程视图

- Android App 中的 WebView 前端通过 HTTPS 调用 REST API。
- Android App 登录后建立 WebSocket 连接。
- API 服务负责鉴权、业务校验、数据库读写。
- 实时消息服务向会话成员推送新消息、已读状态、交换卡片状态变化。
- 后续审核服务可异步处理举报、图片/文本审核。

### 4.4 部署视图

MVP 部署建议：

- App 前端：React/Vite 构建 `dist`，通过 Capacitor 封装为 Android APK。
- 备用 H5：静态资源可部署到 Ubuntu 服务器或 Nginx，作为无法安装 APK 时的备用演示入口。
- API 服务：Node.js 服务部署到 Ubuntu 云服务器。
- 数据库：PostgreSQL 托管服务或 Ubuntu 云服务器自建。
- WebSocket：与 API 服务同进程部署，后续再拆分。
- 媒体资源：原型阶段本地 mock，正式阶段迁移到对象存储。

## 5. 编程规范

### 5.1 TypeScript/React 规范

采用 TypeScript + React 官方推荐实践，并参考 Google TypeScript Style Guide。

核心约定：

- 业务数据必须定义类型，不使用 `any` 绕过领域模型。
- 组件使用 PascalCase 命名，hook 使用 `useXxx` 命名。
- 纯工具函数放入 `lib/`，页面业务状态放入 `hooks/`。
- 页面组件不直接散落 API 请求，优先通过 hook 或 service 调用。
- 表单输入必须进行前端校验，后端仍要重复校验。
- 异步请求统一处理 loading、error、empty 状态。
- 重要业务字段使用 `userId`、`postId`、`cardId` 等稳定 ID，不依赖昵称匹配。

### 5.2 后端规范

- Controller 只处理请求参数和响应，不直接写复杂业务逻辑。
- Service 负责业务规则、权限判断和事务。
- Repository/ORM 层负责数据库访问。
- API 返回统一格式：`data`、`error`、`requestId`。
- 所有写操作必须校验当前登录用户权限。
- 数据库表使用 `createdAt`、`updatedAt`，软删除场景使用 `status` 或 `deletedAt`。
- WebSocket 事件命名使用 `domain:event`，例如 `message:new`、`exchange-request:updated`。

### 5.3 Git 规范

- 分支建议：`feature/auth-api`、`feature/meal-card-api`、`feature/chat-prototype`。
- commit message 建议：`feat: add meal card api prototype`、`docs: add architecture views`。
- 每个功能分支至少包含文档、代码、测试或验证说明。
- 合并前执行 `npm run check`、`npm run build`。

## 6. 关键算法设计

### 6.1 约饭卡推荐匹配算法

详见 `web/docs/11-matching-recommendation-algorithm.md`。该文档作为后续 `matchScore`、`reason`、行为日志、离线评估和模型演进的权威方案。

核心思路从单纯标签匹配升级为“找搭子双向匹配 + 内容平台行为反馈 + 即时约饭场景”的混合推荐：

```text
finalScore =
  reciprocalScore * 0.25 +
  sceneScore      * 0.22 +
  interestScore   * 0.20 +
  behaviorScore   * 0.13 +
  trustScore      * 0.08 +
  qualityScore    * 0.07 +
  freshnessScore  * 0.03 +
  explorationScore* 0.02 -
  penalty
```

设计原则：

- 双向匹配：不仅判断“我是否喜欢这张卡”，还估计“对方是否可能接受我的邀请”。
- 场景优先：饭点、地点、人数和当前活跃状态在约饭场景中权重高于泛兴趣。
- 行为学习：曝光、跳过、详情、主页、关注、邀请、接受、拒绝等行为逐步反哺排序。
- 安全底线：拉黑、举报、高风险内容和重复打扰优先过滤或强降权。
- 可解释输出：服务端应生成用户能理解的中文 `reason`，例如饭点地点接近、标签相似、人数偏好一致。
- 平滑演进：第一版使用 TypeScript 规则排序；数据积累后再复现 LightFM、implicit、双塔召回或学习排序。

### 6.2 “想一起吃”交换卡片算法

目标：点击候选卡片后，生成一条可追踪的约饭邀请，并作为会话消息展示。

流程：

1. 校验当前用户已登录。
2. 校验目标卡片仍处于 `active` 状态。
3. 查找当前用户最近有效约饭卡；如没有，引导用户先发布或生成临时意向卡。
4. 创建或复用双方会话。
5. 创建 `exchange_request`，状态为 `pending`。
6. 写入一条 `message`，类型为 `meal_card_exchange`。
7. 通过 WebSocket 推送给接收方。
8. 接收方选择接受或拒绝后，更新 request 状态并推送 `exchange-request:updated`。

### 6.3 搜索排序算法

搜索对象包括用户、约饭卡、帖子。

MVP 规则：

- 关键词命中标题、正文、昵称、标签、地点。
- 完全匹配优先于部分匹配。
- 当前活跃内容优先于历史内容。
- 已拉黑用户、已删除内容不进入结果。

排序公式：

```text
searchScore = textMatchScore * 0.60 + freshnessScore * 0.20 + interactionScore * 0.20
```

## 7. 技术亮点

1. 从原型 hooks 平滑迁移到真实 API：现有 `useMealCards`、`useCommunityState`、`useExchangeRequests` 已经把业务状态集中起来，后续主要替换 hook 内部实现。
2. 推荐结果可解释：不仅给出匹配分，还输出推荐理由，便于用户理解和评审演示。
3. 聊天与约饭邀请融合：把“想一起吃”建模为消息类型和交换请求，而不是一次性页面跳转。
4. 数据模型预留正式 ID：逐步从昵称匹配迁移到 `userId`、`authorId`、`conversationId`。
5. App 交付路径清晰：React/Vite 移动端前端可通过 Capacitor 快速封装 APK，同时后端 API 与领域模型不绑定具体客户端。
6. 风险前置：实时聊天、内容审核、媒体上传、推荐冷启动都在架构中预留扩展点。

## 8. 技术风险与缓解

| 风险 | 影响 | 缓解措施 |
| --- | --- | --- |
| 昵称匹配导致数据错乱 | 关注、聊天、详情跳转不可靠 | 正式接口全部使用稳定 ID |
| 实时聊天实现复杂 | 评审演示可能不稳定 | 技术原型先完成单机 WebSocket demo，再扩展多用户 |
| 推荐算法缺少真实数据 | 匹配效果难证明 | 使用可解释规则算法 + mock 行为数据验证 |
| 媒体上传与审核成本高 | 社区功能不完整 | MVP 用资源 URL/mock，先完成帖子主流程 |
| 前端页面状态与真实路由不一致 | 刷新、分享、回退体验差 | 第二轮逐步引入 react-router 动态路由 |
| App 打包和 Android 网络限制 | APK 能打开但无法请求接口 | 提前接入 Capacitor，API 使用服务器 IP/域名，必要时配置 HTTPS 或明文网络 |
| 举报审核流程未闭环 | 安全风险 | 先入库 reports，后台审核作为后续扩展 |
