# 02 页面跳转与 Use Case 拆解

本文件是 [prototype-navigation-usecases.md](./prototype-navigation-usecases.md) 的模块化拆解。完整 Mermaid 总图和迁移跳转规则以 `prototype-navigation-usecases.md` 为准。

## 底部导航

| 导航项 | 当前页面 | 主要动作 |
| --- | --- | --- |
| 首页 | `Home` | 划卡、筛选、搜索、发布约饭卡入口、想一起吃 |
| 社区 | `Community` | 浏览帖子、发布帖子、帖子详情、评论、媒体查看 |
| 发卡片 | `CreateCard` | 创建约饭卡、选择头像、选择/创建标签、发布 |
| 消息 | `Chat` | 消息列表、消息搜索、聊天详情、交换卡片 |
| 我的 | `Profile` | 头像、偏好标签、我的内容、关注用户、设置入口 |

## 全局搜索跳转

搜索浮层从首页和社区打开。搜索结果包括：

- 用户：打开用户主页详情。
- 约饭卡片：打开约饭卡片详情。
- 帖子：打开帖子详情。

当前实现中，搜索详情浮层叠在搜索浮层上。关闭详情后仍回到搜索页，这是为了保留搜索上下文。
用户主页详情里可以继续点击该用户发布的帖子和约饭卡片，分别进入帖子详情和约饭卡详情。
非本人用户主页显示“关注”按钮；关注后会同步到“我的”页关注用户列表和消息页“新增关注”入口。

搜索浮层的来源页目前不会显式记录在数据结构里，而是通过浮层叠放自然保留。迁移时如果搜索变成独立页面，需要用 route state 或 query 记录来源页。

后续动态路由建议：

- 用户详情：`/users/:userId`
- 约饭卡详情：`/cards/:cardId`
- 帖子详情：`/posts/:postId`
- 搜索结果详情返回：使用路由栈或 modal route，而不是丢失搜索上下文。

## 首页跳转

| 用户动作 | 当前结果 | 后续建议 |
| --- | --- | --- |
| 点搜索 | 打开 `SearchOverlay` | 打开搜索页或搜索 modal |
| 点发布约饭卡 | 进入 `CreateCard` | 路由 `/cards/create` |
| 多选标签 | 本页筛选卡片 | URL 可选同步 query |
| 左右滑卡 | 切换卡片 | 手势事件按平台适配 |
| 点想一起吃 | 生成交换请求并进入聊天详情 | POST exchange request 后 deep link 到会话 |

“想一起吃”和底部“消息”必须分开处理：前者是 deep link 到聊天详情，后者永远进入消息列表。

## 社区跳转

| 用户动作 | 当前结果 | 后续建议 |
| --- | --- | --- |
| 点帖子 | 打开 `Community` 内完整帖子详情 | 路由 `/posts/:postId` |
| 点帖子作者头像 | 打开作者用户主页详情 | 路由 `/users/:userId` |
| 点照片 | 打开大图浮层 | 统一媒体 viewer |
| 点视频 | 打开沉浸式视频详情 | 真实视频播放器 |
| 点评论 | 打开评论 sheet | 评论路由或详情页内部锚点 |
| 发布帖子 | 插入本地 state 并立即打开新帖详情 | POST `/posts` 后跳新详情页 |

社区内帖子详情比搜索里的帖子详情更完整，含点赞、收藏、评论 sheet 和媒体沉浸视图。后续应合并成同一个帖子详情页。

## 消息跳转

- 点击底部“消息”：永远打开消息列表。
- 从首页“想一起吃”进入：自动打开对应用户聊天详情一次。
- 聊天详情返回：回到消息列表。
- 消息搜索：只搜索消息页内部会话/群聊/记录，不等同于全局搜索。
- 点“赞和收藏”：打开通知列表，可进入被点赞/收藏的帖子详情。
- 点“新增关注”：打开关注通知列表，可进入关注用户主页。
- 点“评论和@”：打开评论/@列表，可进入相关帖子的评论区。

当前用 `autoOpenRequestId` 和 `chatListResetSignal` 区分“深链进聊天”和“底部导航进消息列表”。正式迁移时应改为导航参数。

## 我的与设置跳转

- 我的头像：打开头像查看/更换弹层。
- 我的偏好：打开标签编辑弹层。
- 我的内容列表：可打开帖子、卡片、评论对应详情。
- 我的关注用户列表：包含原始关注用户和新关注用户，可打开用户主页。
- 设置：进入设置页。
- 设置列表项：在设置页内部切换二级详情。
- 点“我发布的评论”：打开相关帖子详情并定位到评论区语义。
- 点“关注用户”：打开其他用户主页详情。

后续建议：

- 设置可保持单页内部状态，也可拆成 `/settings/:section`。
- 用户、帖子、卡片详情建议统一到动态详情页，不要长期维护搜索详情和社区详情两套展示。
