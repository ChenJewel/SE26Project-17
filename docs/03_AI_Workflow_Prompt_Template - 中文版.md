# Ueat AI开发工作流提示词模板

## 1. 文档目的

本文档定义 Ueat 项目中使用 AI 工具（如 Codex / ChatGPT / 其他 AI 助手）时的标准化提示词结构。

它用于确保：
- 所有 AI 生成内容保持一致
- 输出与项目文档保持一致
- 防止 AI 生成无依据或不符合项目范围的功能
- 建立结构化的软件工程 AI 辅助开发流程

---

## 2. 核心原则

所有 AI 任务必须遵循以下结构：

> Base Context（固定）+ Task（可变）

- Base Context：定义项目背景、项目定位、规则与约束。
- Task：定义当前具体要完成的工作。

---

## 3. 基础提示词（固定部分，不得修改）

以下内容必须在所有 AI 任务中使用。

```text
你正在 Ueat（校园约饭搭子系统）项目中工作。

## 项目定位
Ueat 以“约饭搭子”作为校园低压力线下互动的初始入口。

项目目标并不只是安排一顿饭，而是通过共同吃饭这一自然、低压力的场景，帮助学生缓解孤单、认识新同学、建立有意义的校园连接，并在后续 Use Case 中逐步支持见面后的继续互动。

Ueat 不是：
- 约会软件
- 泛社交平台
- 纯聊天应用
- 只在“吃完一顿饭”后就结束的工具

Ueat 是：
- 面向校园场景的线下互动系统
- 降低陌生同学第一次见面门槛的结构化工具
- 可在 /docs、用户研究与后续用例支持下，逐步扩展信任、安全、反馈、后续互动与长期连接能力的系统

## 资料来源（Source of Truth）
必须严格以 /docs 目录作为唯一事实来源。
禁止凭空编造信息或假设未提供的内容。

关键文件：
- 00_Project_Context.md
- 01_Pre-Full_Conversation_summary.md
- 02_Questionnaire.md
- 03_AI_Workflow_Prompt_Template.md
- 04_Archive.md

## 规则约束
- 必须保持与现有项目范围一致
- 不允许引入未在 /docs 中支持的功能
- 所有设计必须符合软件工程系统级视角
- 避免幻想式或不合理扩展
- 必须把“约饭匹配”视为入口场景，而不是产品最终边界
- 任何超出约饭匹配的功能扩展，都必须由 /docs、用户研究或后续 Use Case 文档支撑
- 不得把 Ueat 扩展成约会软件、泛社交平台或纯聊天产品
```

---

## 4. 任务模板（可变部分）

在基础提示词之后，使用以下结构描述具体任务。

```text
## Task
{在这里描述具体任务。}

## Input Materials
{在这里粘贴用户提供的额外材料、草稿、需求、附件或说明；如果没有，则写“无”。}

## Expected Output
{在这里描述期望输出格式，例如 Vision 文档、PRD、Use Case、UI 页面结构、架构设计、实现计划或 Codex 开发 Prompt。}

## Constraints
- 输出必须以 /docs 作为唯一事实来源
- 输出必须保持在 Ueat 项目范围内
- 输出不得引入无文档依据的功能
- 信息不足时必须明确标注，不得编造
- 所有假设都必须显式说明

## Quality Checklist
- 与 Ueat 项目定位一致
- 可复用、可模块化扩展
- 具备工程可用性
- 适合软件工程协作
- 可直接用于 Codex / ChatGPT / AI 工具
```
